Telegram Widget — Product Requirements Document

Status: Draft Author: HyperDesk Agent Type: Widget + Bridge Daemon Distribution: ZenBin Install Risk: Medium

Source PRD: zenbin.org/p/telegram-widget-prd

Definition

A HyperDesk Telegram Widget is a user-installable widget that pairs a sandboxed HTML chat UI (right panel) with a local Telegram bridge daemon (Node.js + grammy). The daemon polls the Telegram Bot API and relays messages to/from HyperDesk's agent dispatch system via WebSocket. Users interact with their Pi agent sessions from Telegram; the widget provides live visibility and configuration.

Not a bot platform, a multi-user service, or a hosted SaaS. Every installation is fully independent, per-user, and runs entirely on the user's machine.

Goals

  • Users can chat with their HyperDesk Pi agent from Telegram
  • Widget provides a live chat view, config, and status
  • Zero-install dependency model — grammy bundled with the widget
  • Shareable via ZenBin for any HyperDesk user to install

Non-Goals

  • Webhook mode (polling only)
  • File/media sharing between Telegram and HyperDesk
  • Voice messages or Telegram voice notes
  • Persistent message history across restarts
  • Auto-update system

Constraints & Security

  • Bot token stored in HyperDesk profile secrets; never logged or echoed
  • Chat ID allowlist validates incoming messages before forwarding
  • WebSocket binds to 127.0.0.1 only — never exposed to network
  • No cross-user data leakage — zero shared state between installations
  • command.run requires approval — mitigation: session-wide allow for bridge command
  • Telegram 4096 char limit — bridge splits long responses at paragraph boundaries

Resolved Open Questions

  1. Conversation memory: Yes — bridge maintains history, includes it in each agent.dispatch prompt
  2. Streaming approach: Send chunks — new message per chunk (simpler, noisier)
  3. Multi-chat support: Yes — v2 scope (multi-user)
  4. Command routing: Agent — Telegram commands dispatched to the agent, not handled by bridge

Architecture

Telegram ──Bot API──→ Telegram Bridge Daemon (Node.js + grammy)
                          │
                          │  WebSocket (localhost:18765)
                          │
                    HyperDesk Widget (right panel)
                          │
                          │  widget.agent.dispatch()
                          │
                    Pi Agent (background session)

Step Plan

Step 1: Manifest + Widget Skeleton

Acceptance criteria:

  • hyperdesk.extension.json with correct schema, id, name, permissions (agent.dispatch, command.run, secrets.read:telegram), widget declaration (right-panel, entry point, toggle command)
  • ui/index.html basic skeleton with three tabs (Chat, Config, Status)
  • SDK files included: ui/hyperdesk-widget.js, ui/hyperdesk-widget.d.ts

Evidence: Manifest validates against HyperDesk widget schema; file structure matches spec

Step 2: Bridge Daemon (telegram-bridge.cjs)

Acceptance criteria:

  • Node.js process using grammy for Telegram Bot API long-polling
  • WebSocket server on localhost (configurable port, auto-increment on conflict)
  • Commands: /start, /stop, /status, /help
  • Any non-command text forwarded to widget via WebSocket
  • Chat ID allowlist validation (silently drops unmatched)
  • Error handling: invalid token, rate limit backoff, WebSocket reconnect (5s), port conflict
  • package.json with grammy + all transitive deps bundled

Evidence: node scripts/telegram-bridge.cjs --help runs; grammy import resolves from bundled node_modules

Step 3: Widget UI — Chat Tab

Acceptance criteria:

  • Telegram-styled message bubbles (blue right for agent, gray left for user)
  • Timestamps on messages
  • "Agent is typing..." indicator while streaming
  • Auto-scroll to bottom on new messages
  • Receives incoming Telegram messages via widget.onCommand
  • Dispatches agent messages via widget.agent.dispatch()
  • Streams agent output (hyperdesk:event) to bridge via WebSocket

Evidence: Widget loads in HyperDesk panel; tabs render; theme tokens applied

Step 4: Widget UI — Config + Status Tabs

Acceptance criteria:

  • Config tab: Bot Token (password input), Chat ID (optional, allowlist), Start Bridge / Stop Bridge buttons
  • Status tab: bridge process info, last message timestamp, event log (in/out counts)
  • Connection status indicator (running / stopped / error)
  • Config persists across widget reloads (HyperDesk settings API)
  • Start/Stop buttons call command.run with correct arguments and secret env injection

Evidence: Config form renders; start triggers approval prompt with correct command

Step 5: Data Flow Integration

Acceptance criteria:

  • Incoming: Telegram → bridge → WebSocket → widget.onCommand → widget.agent.dispatch()
  • Outgoing: agent dispatch → hyperdesk:event deltas → widget collects → WebSocket → bridge → Telegram sendMessage/editMessageText per chunk
  • Conversation history maintained in bridge, prepended to each dispatch prompt
  • Round-trip: message sent to bot appears in widget chat view
  • Bridge survives widget page reload (daemon is independent process)

Evidence: Full end-to-end test: message on Telegram → appears in widget → agent responds → chunked back to Telegram

Step 6: README + Distribution Artifacts

Acceptance criteria:

  • README.md with install steps, first-run config, permission explanations, troubleshooting
  • File structure matches spec (all files in place)
  • Packageable with scripts/package-widget.cjs (if available) or manual bundle check
  • File digests (SHA-256) included in README for verification

Evidence: README is complete; all required files exist; bundle would package cleanly
