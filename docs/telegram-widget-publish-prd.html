Telegram Widget — Publish & Test Plan

Task: Package the tg-widget for public installation, create a GitHub repo at twilson63/telegram-widget, and write feature coverage tests that validate all bridge + widget functionality.

Definition

Feature coverage tests = automated Node.js tests (no browser needed) that verify the telegram-bridge.cjs daemon's core logic: message routing, chat allowlist enforcement, token secret handling, message chunking, conversation history, command parsing, and WebSocket handshake/protocol. The tests use zero external network calls — they mock the Telegram Bot API and test via the localhost WebSocket and internal functions.

What this is NOT: Not end-to-end tests with a real Telegram bot (those require a bot token and network). Not widget UI tests (those need a browser). Not integration tests with HyperDesk. This is unit + integration test coverage of the bridge daemon's logic.

Goals

  1. Create test suite covering all bridge daemon features
  2. Set up GitHub repo twilson63/telegram-widget with source + tests
  3. Update widget ID from tom.telegram to twilson63.telegram
  4. Publish a ZenBin share page for one-command installation
  5. Update README with test instructions

Constraints

  • Tests must not require real credentials, network, or a running HyperDesk instance
  • Tests must be runnable with npm test from the scripts/ directory
  • Token secrets must never appear in test output or logs
  • No browser automation needed for tests (bridge is Node-only logic)

Non-goals

  • End-to-end Telegram + HyperDesk integration testing
  • UI/component tests for the widget HTML
  • Performance or load testing
  • Multi-user Telegram bot architecture (v2 scope)

Step-by-step Plan

Step 1: Test Infrastructure

    - Test runner set up: scripts/package.json has "test": "node --test" (Node 22 native test runner) run cd scripts && npm test and verify exit 0 with no tests yet
    - Test helper module created: scripts/test/test-helper.cjs exports utility functions for WS client, mock context, and assertion helpers file exists and exports are importable
    - .gitignore excludes scripts/node_modules/, *.log, test fixtures with tokens git status shows clean node_modules

Step 2: Bridge CLI & Config Tests

    - Without token and without --widget-managed, exit code is 1 spawn node, verify exitCode === 1 and stderr contains "required"
    - --help prints usage and exits 0 spawn node, verify stdout contains "telegram-bridge"
    - --version prints version and exits 0 spawn node, verify stdout contains "v0.2.0"
    - TELEGRAM_BOT_TOKEN env var is accepted without --widget-managed spawn with env, verify it starts without "ERROR" stderr

Step 3: WebSocket Protocol Tests

    - Bridge starts WS server on localhost connect to port 18765, receive bridge:status message
    - Widget sends bridge:configure with token → bridge starts bot polling send config over WS, verify status changes to "configuring" then "running"
    - Widget sends bridge:stop → bridge stops polling send stop, verify status changes to "stopped"
    - Widget sends bridge:ping → receives bridge:status send ping, verify status response
    - Port auto-increment when 18765 is occupied start two instances, verify second uses 18766

Step 4: Message Routing Tests

    - Bridge sends telegram:message to widget on incoming Telegram message mock ctx.on("message:text"), verify WS sends correct payload with chatId
    - Agent response via telegram:message from widget → bridge sends to Telegram send telegram:message payload over WS, verify bot.api.sendMessage called with correct chatId
    - Long responses are split into ≤3500 char chunks send 10KB text, verify 3+ sendMessage calls with correct chunk sizes
    - Unauthorized chat ID (with allowlist) → message silently dropped set allowlist, send from different chatId, verify no WS message sent

Step 5: Telegram Commands Tests

    - /start → sends welcome message with chatId/port, sets polling active mock ctx, verify api.sendMessage called
    - /stop → stops polling mock ctx while polling, verify stop called and status sent to widget
    - /status → sends bridge status summary mock ctx, verify api.sendMessage with status text
    - /help → lists available commands mock ctx, verify api.sendMessage with help text

Step 6: History & Security Tests

    - Conversation history bounded to 40 messages max (20 exchanges) send 25 user messages, verify history.length ≤ 40
    - Bot token never appears in command line, env, or logs grep bridge code: token only flows via WS, never via args/env in --widget-managed mode
    - Invalid token (401) → error sent to widget, polling stops mock bot.catch with 401 error, verify widget gets error status

Step 7: Package & Publish

    - Widget ID updated to twilson63.telegram in manifest cat hyperdesk.extension.json | grep id
    - GitHub repo created and pushed repo exists at github.com/twilson63/telegram-widget
    - ZenBin share page published URL returned from zenbin_publish
    - README updated with test instructions and badge README has npm test section
