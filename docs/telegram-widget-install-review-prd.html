Telegram Widget Install + Bridge Review

Status: implementation review

Project-specific definition

“Installing the widget” means HyperDesk has a validated widget manifest and required assets under ~/.hyperdesk/widgets/tom.telegram/, so View → Widgets → Telegram can load the sandboxed right-panel iframe. It does not mean merely keeping a source checkout in ~/desk-projects/tg-widget.

“Bridge running” means a local Node daemon is listening on 127.0.0.1:18765, is configured with a Telegram bot token, and routes Telegram text to the widget over WebSocket. A listening daemon alone is insufficient if the widget is not installed or if the widget ignores the daemon’s telegram:message events.

Goals

  • Find why the widget is not appearing/installing in HyperDesk.
  • Verify whether the bridge process is running and which copy it is running from.
  • Fix bridge/widget integration bugs that would make a running bridge look connected but not dispatch messages.
  • Preserve token privacy: no token in command line, process list, logs, or chat.

Non-goals

  • No Telegram live-token test without the user’s bot token.
  • No public hosting or webhook mode.
  • No broad HyperDesk app-source changes.

Research evidence

  • Read widgets.md: widgets must live under ~/.hyperdesk/widgets/ and bridge calls require declared permissions.
  • Local check showed ~/.hyperdesk/widgets/tom.telegram did not exist.
  • lsof showed a Node bridge listening on 127.0.0.1:18765, PID 82808, cwd /Users/tron/desk-projects/tg-widget.
  • WebSocket smoke check returned {"type":"bridge:status","payload":{"port":18765,"status":"running"}}.
